# JumpLifter -- CE Compatibility Patch

| Field          | Value                                        |
|----------------|----------------------------------------------|
| **Mod**        | JumpLifter by rabiosus                       |
| **Steam ID**   | 3493717994                                   |
| **packageId**  | `rabiosus.jumplifter`                        |
| **Mod type**   | XML-only                                     |
| **Patch file** | `Patches/JumpLifter.xml`                     |
| **RimWorld**   | 1.5, 1.6 (version-agnostic defName target)  |

---

## 1. What Broke

JumpLifter adds `Mech_JumpLifter`, a light mechanoid that teleports cargo via
an AutoBlink comp. It uses `Mech_Light` body, body size 0.7, MoveSpeed 5.2.
It is a non-combat hauler -- a faster, teleporting cousin of the vanilla Lifter.

Combat Extended patches all six vanilla light mechs individually by defName.
Each receives two things CE needs to function correctly:

- **`CompProperties_ArmorDurability`** -- governs how mech armor degrades
  under sustained fire, regenerates passively, and can be repaired by colonists.
- **`RacePropertiesExtensionCE`** (`bodyShape`) -- tells CE's projectile
  system what collision geometry to use for the pawn's body when calculating
  hit chances on downed or standing targets.

CE has no awareness of JumpLifter, so `Mech_JumpLifter` gets neither. Two
concrete failures result:

### Gap 1: No Armor Durability

Every other light mech in CE has a durability pool. Under fire, their armor
degrades toward `MinArmorPct`, eventually softening them up. Without the comp,
JumpLifter's armor *never* degrades. It silently becomes the most durable
light mech in the game -- not because it was designed to be, but because the
degradation system has no handle on it.

### Gap 2: Undefined Body Shape

When CE resolves collision factors for a pawn without `RacePropertiesExtensionCE`,
it falls back to `BodyType.Undefined`, which maps to the `Invalid` shape
(collision factors 1/1/1/1 on all axes). CE logs:

```
ErrorOnce("CE returning BodyType Undefined for pawn ...")
```

The correct shape for a bipedal `Mech_Light`-body mechanoid is `Humanoid`,
which provides realistic collision factors:

| Factor        | Invalid (fallback) | Humanoid (correct) |
|---------------|-------------------:|-------------------:|
| width         |                1.0 |               0.25 |
| height        |                1.0 |                1.0 |
| widthLaying   |                1.0 |                0.5 |
| heightLaying  |                1.0 |                0.3 |

With `Invalid`, a downed JumpLifter presents a 1.0 x 1.0 target instead of
the 0.5 x 0.3 silhouette every other light mech presents when down.

## 2. Design Problem and Options Considered

The goal: bring `Mech_JumpLifter` into parity with CE's treatment of vanilla
light mechs without over- or under-tuning for its actual role and mobility.

### Durability Tier Selection

CE assigns durability by mech role and body size:

| Mech           | Durability | Notes               |
|----------------|------------|----------------------|
| Cleansweeper   |        500 | Smallest light mech  |
| Militor        |        700 | Standard light tier  |
| Lifter         |        700 | Standard light tier  |
| Constructoid   |        700 | Standard light tier  |
| Paramedic      |        700 | Standard light tier  |
| Fabricor       |        700 | Standard light tier  |
| WarUrchin      |       1000 | Combat-hardened      |

JumpLifter is body size 0.7, non-combat, hauler-role -- it slots into the
standard 700 tier. No case for 500 (not undersized) or 1000 (not combat).

### Body Shape Selection

`Mech_JumpLifter` uses the `Mech_Light` body -- two legs, bipedal posture,
identical skeletal structure to the vanilla Lifter. The only valid CE body
shape for this geometry is `Humanoid`.

### Carry Weight, Bulk, and Combat Stats

CE's parent abstraction for light mechs sets CarryWeight 50 and CarryBulk 20.
The vanilla Lifter gets an override to 60/25 because its MoveSpeed is only 2.8
-- CE compensates for low mobility with higher per-trip capacity.

JumpLifter has MoveSpeed 5.2 plus AutoBlink teleportation, giving it
dramatically superior hauling throughput. Adding Lifter's carry override would
double-buff it. The parent values of 50/20 are correct.

No melee stat overrides are needed. JumpLifter is non-combat; the parent
values inherited from the CE light mech abstraction are appropriate.

## 3. Implementation

The entire patch is a single file: **`Patches/JumpLifter.xml`**.

### Patch Structure

The outer `PatchOperationFindMod` gates on both `Combat Extended` and
`JumpLifter` being active. Inside, a `PatchOperationSequence` runs two
operations in order.

### Op 1: ArmorDurability Comp (lines 11-29)

```xml
<li Class="PatchOperationAdd">
    <xpath>Defs/ThingDef[defName="Mech_JumpLifter"]/comps</xpath>
    <value>
        <li Class="CombatExtended.CompProperties_ArmorDurability">
            <Durability>700</Durability>
            <Regenerates>true</Regenerates>
            <RegenInterval>1250</RegenInterval>
            <RegenValue>5</RegenValue>
            <Repairable>true</Repairable>
            <RepairIngredients>
                <Steel>5</Steel>
                <Plasteel>5</Plasteel>
            </RepairIngredients>
            <RepairTime>300</RepairTime>
            <RepairValue>200</RepairValue>
            <MinArmorPct>0.5</MinArmorPct>
        </li>
    </value>
</li>
```

`PatchOperationAdd` appends to the existing `<comps>` list. This works because
JumpLifter already declares a `<comps>` block (AutoBlink comp); no conditional
node creation is needed.

Regen: 5 HP every 1250 ticks (~20.8 seconds at 1x), matching standard light
mech rates. Repair: 5 Steel + 5 Plasteel per action, restoring 200 durability
over 300 ticks. `MinArmorPct` 0.5 floors armor at half strength when depleted.

### Op 2: RacePropertiesExtensionCE (lines 31-38)

```xml
<li Class="PatchOperationAddModExtension">
    <xpath>Defs/ThingDef[defName="Mech_JumpLifter"]</xpath>
    <value>
        <li Class="CombatExtended.RacePropertiesExtensionCE">
            <bodyShape>Humanoid</bodyShape>
        </li>
    </value>
</li>
```

`PatchOperationAddModExtension` auto-creates the `<modExtensions>` node if it
does not already exist on the ThingDef. This is safer than a raw
`PatchOperationAdd` targeting the xpath, and future-proof against JumpLifter
adding its own mod extensions.

### What Is Not Patched

The patch deliberately omits several things that might seem like candidates:

- **No CarryWeight/CarryBulk override.** The parent values (50/20) are correct
  for a teleporting hauler. See the design rationale in Section 2.
- **No melee stat overrides.** Non-combat unit; parent values are appropriate.
- **No weapon patches.** JumpLifter carries no weapons.
- **No `<comps>` node creation.** JumpLifter already has one (AutoBlink).
- **No version-specific branching.** The patch targets `defName`, which is
  stable across RW 1.5 and 1.6.

## 4. Why This Approach

### Why Not a Wildcard Patch on All Light Mechs

CE patches vanilla light mechs individually by defName because durability
tiers differ (Cleansweeper 500, standard 700, WarUrchin 1000). A wildcard
would flatten these distinctions. V2CEPatch follows the same convention.

### Why PatchOperationAddModExtension Over PatchOperationAdd

JumpLifter currently has no `<modExtensions>` block. A raw `PatchOperationAdd`
targeting that xpath would fail with a null node error.
`PatchOperationAddModExtension` handles this gracefully and future-proofs
against JumpLifter adding its own mod extensions later.

### Why a Single File, Not a Folder Structure

JumpLifter is an XML-only mod adding one mechanoid. The patch requires exactly
two operations. There is no need for Defs, C# assemblies, or multi-file
organization. A single patch file under `Patches/` is the minimal correct
structure.

### Why No Carry Override Despite Lifter Having One

This is the most likely point of confusion for reviewers. The vanilla Lifter
gets CarryWeight 60 / CarryBulk 25 (versus the parent's 50/20) because its
MoveSpeed 2.8 makes it the slowest light mech. CE compensates with higher
per-trip capacity to keep total hauling throughput reasonable.

JumpLifter at MoveSpeed 5.2 plus AutoBlink teleportation already far exceeds
a carry-boosted Lifter's throughput. The parent 50/20 values keep the tradeoff
honest: lower per-trip capacity, dramatically higher trip frequency.

## 5. Post-Patch Stat Summary

After patching, `Mech_JumpLifter` presents these CE-relevant stats:

| Stat              | Value                  | Source                          |
|-------------------|------------------------|---------------------------------|
| ArmorRating_Sharp | 2                      | Inherited from CE LightMechanoid|
| ArmorRating_Blunt | 3                      | Inherited from CE LightMechanoid|
| MoveSpeed         | 5.2                    | JumpLifter original def         |
| CarryWeight       | 50                     | Inherited from CE parent        |
| CarryBulk         | 20                     | Inherited from CE parent        |
| ArmorDurability   | 700                    | **Patch Op 1**                  |
| Regen             | 5 HP / 1250 ticks      | **Patch Op 1**                  |
| Repair cost       | 5 Steel + 5 Plasteel   | **Patch Op 1**                  |
| Repair value      | 200 HP per action      | **Patch Op 1**                  |
| RepairTime        | 300 ticks              | **Patch Op 1**                  |
| MinArmorPct       | 0.5                    | **Patch Op 1**                  |
| bodyShape         | Humanoid               | **Patch Op 2**                  |

JumpLifter slots into the non-combat hauler tier alongside the vanilla Lifter:
same durability (700), same parent armor (2/3 Sharp/Blunt), bipedal Humanoid
body shape, lower per-trip carry, dramatically higher mobility.

## 6. Testing Notes

1. Load a game with Combat Extended, JumpLifter, and V2CEPatch active.
2. Spawn `Mech_JumpLifter` via dev mode.
3. Confirm the ArmorDurability comp is visible in the inspect pane.
4. Verify debug log has no `CE returning BodyType Undefined for pawn` errors
   referencing JumpLifter.
5. Down the mech and confirm projectile calculations use Humanoid collision
   factors (0.5 widthLaying, 0.3 heightLaying), not the Invalid fallback.
